## 🚨 CRITICAL: Anti-Waste Rules (必须遵守)

### 事件背景
2026-05-27: `arrived-vehicle-monitor` boulder 完成后，Atlas 因系统 nudge 陷入 20+ 次重复 `todowrite` 循环，浪费大量时间/token。

### 铁律

#### 1. 先读状态，再决定行动
收到任何 `[internal]` nudge 或恢复消息时：
- **首先**检查 `.omo/boulder.json` 的 `status`
- 如果 `status === "completed"` 且 `elapsed_ms > 0` → **直接输出最终总结，不再调用任何工具**
- ❌ 禁止：重新 read → 重新 todowrite → 重复确认

#### 2. Completed = 停止
- 如果 todo 已经是 `completed`，**禁止再次调用 todowrite**
- 如果 boulder 已经是 `completed`，**禁止启动新任务或验证**
- 唯一允许的操作：输出最终总结并等待用户新指令

#### 3. 区分消息类型
| 消息类型 | 处理方式 |
|---------|---------|
| `[internal] Continue...` | 检查状态，无变化则简短回应或不回应 |
| `BOULDER COMPLETE` nudge | 输出最终总结，停止，不调用工具 |
| 用户新指令 | 正常工作流 |

#### 4. 最终总结 = 终点
- 输出最终总结后，明确说"等待你的下一步指令"
- 之后不再主动调用任何工具，除非用户发送新消息

### 行动前自检
- [ ] 这个操作会改变任何状态吗？如果不会，跳过。
- [ ] 这个 todo 已经是 completed 了吗？如果是，跳过。
- [ ] Boulder 已经是 completed 了吗？如果是，只输出总结。

## 🏗️ Architecture Patterns

### Backend (Spring Boot)
- **Controllers**: REST endpoints, parameter validation, return `ResponseEntity<ApiResponse<T>>`
- **Services**: Business logic, data aggregation, calculator classes
- **DTOs**: Immutable record-like classes for API responses
- **Tests**: JUnit 5, located in `src/test/java` mirroring main structure

### Frontend (Vue 3 + Element Plus + ECharts)
- **Views**: Dashboard-level pages (e.g., `Dashboard.vue`, `ArrivedDashboard.vue`)
- **Components**: Tab-specific chart components (e.g., `OverallTab.vue`, `SegmentTab.vue`)
- **Routing**: Vue Router with nested routes under `/dashboard/`
- **Charts**: ECharts via `vue-echarts`, configs in `getChartOption()` methods

## 📊 Domain Model

### Vehicle Transport Status
| Status | Meaning | Time Fields |
|--------|---------|-------------|
| `IN_TRANSIT` | 在途 | `actualDepartureTime` set, `arriveShopTime` null |
| `ARRIVED` | 已到店 | `arriveShopTime` != null |

### Delay Categories (Efficiency Buckets)
| Category | Ratio | Color |
|----------|-------|-------|
| `EFFICIENT` | < 80% | Green |
| `NORMAL` | 80% - 100% | Yellow |
| `DELAYED` | > 100% | Red |

Formula: `efficiency = (actualTotalTime / cumulativeOTD) * 100`

## 📝 Work Plan Conventions

- Plans stored in `.omo/plans/{name}.md`
- Tasks use bare-number format: `1.`, `2.` (NOT `T1.` or `Task 1`)
- Final wave tasks: `F1.`, `F2.` (NOT `T-F1.`)
- Each task MUST include QA scenarios with specific selectors/data/assertions

## ⚠️ Known Gotchas

1. **Duplicate files**: Check for duplicate calculator/service files (e.g., `util/` vs `service/` packages)
2. **Field name mismatches**: Frontend must match DTO field names exactly (camelCase)
3. **Build warnings**: ECharts + Element Plus cause large chunk warnings (~1MB each), pre-existing, not regressions
4. **Pre-existing test failures**: `StatusCalculatorTest.java` has compilation issues unrelated to new work
